# v0.42.58.1 migration: durable conversation backfills and schema inheritance

This release changes two runtime defaults and fixes schema-pack inheritance. No
database migration is required.

## What changes automatically

- Bulk conversation backfills default to `meeting,slack`.
- Fresh completed and scanned-not-extractable outcomes are durable even after
  operation-checkpoint cleanup.
- Active packs include declarations inherited through `extends`.
- Cycle phases remain leaf-local and are never inherited.

## Existing installations

1. Restart every long-running gbrain worker after upgrading.
2. If `cycle.conversation_facts_backfill.types` was never set, no config edit
   is required. The new default is already `meeting,slack`.
3. If that key explicitly includes `email`, keep it only when individual
   email pages are intentionally the extraction unit. Otherwise set:

   ```bash
   gbrain config set cycle.conversation_facts_backfill.types '["meeting","slack"]'
   ```

4. Run a serial quality canary:

   ```bash
   gbrain extract-conversation-facts --types meeting,slack --limit 20 --workers 1 --max-cost-usd 0.25 --yes
   ```

5. Inspect sample facts before increasing the cap. Then verify:

   ```bash
   gbrain schema show
   gbrain schema stats --json
   gbrain doctor
   ```

## Email handling

Routine email should flow through thread aggregation or the real-time fact
extractor. `--types email` remains available for an explicit one-off backfill.
The separate scanned-not-extractable outcome prevents a successful negative
scan from being reported as unfinished, but transport, parser, and model
failures remain retryable.
